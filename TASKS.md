- [ ] Sweep watchdog must escalate to `SIGKILL` so a wedged backend cannot run 15× past `TG_SWEEP_MAX` (@devin-session-4)
  - **ID**: sweep-watchdog-escalate-to-sigkill
  - **Tags**: sweep, watchdog, deadline, signals, backend-resilience, AIFN-720-not-applicable
  - **Plan**:
    - [ ] Extend sweep watchdog with SIGINT → SIGTERM → SIGKILL escalation sequence
    - [ ] Add wall-clock hard fallback (`_sweep_start_epoch` + `cap + 60s` cutoff)
    - [ ] Self-heal `wait $! 2>/dev/null || exit 0` so a stray SIGTERM does not disarm the watchdog
    - [ ] Emit `sweep_watchdog escalation=SIGKILL pid=N elapsed=Ns` log marker on the forced-kill path
    - [ ] Add `_FAKE_DEVIN_IGNORE_TERM=1` knob to the fake-backend fixture so tests can simulate the wedged case
    - [ ] Add three regression tests (wedged backend bounded; spurious watchdog SIGTERM; escalation marker emitted)
    - [ ] Update man page TROUBLESHOOTING and docs/architecture.md with the new escalation contract
  - **Source**: `taskgrind-2026-05-01-2245-dotfiles-37908.log` — after `backend_rotated from=devin to=claude-code reason=zero_ship_streak` at 03:56, the launched sweep ran until 11:24 with `sweep_done exit=143 elapsed=26908s cap=1800s`. The cap was 1800 s (30 min); the sweep ran 26 908 s (7 h 28 m), 14.95× over its budget, before something else (apparently the operator) sent `SIGTERM`. The watchdog never fired, so 7 hours of remaining grind budget were burned on a single wedged sweep.
  - **Details**: The sweep watchdog at `bin/taskgrind:4312-4334` has two correctness gaps that combine into the observed 15× overrun:
    1. **No `SIGKILL` escalation**. After the cap expires, the watchdog sends `kill -INT $_dvb_pid`, sleeps 15 s, then sends `kill $_dvb_pid` (default `SIGTERM`). A backend (or any subprocess group) that ignores both signals — claude-code stuck inside an unresponsive HTTP retry loop is one observed case — keeps running, and the watchdog has no further recourse. There is no `kill -KILL` after the `SIGTERM` grace.
    2. **Watchdog dies silently if its own `wait` is interrupted**. The inner `wait $! 2>/dev/null || exit 0` lines exit the watchdog cleanly on any signal interruption. Once the watchdog process exits, nothing else enforces the cap; the parent script's `wait "$_dvb_pid"` blocks forever on the wedged backend. The `trap 'kill $! 2>/dev/null; exit 0' TERM` makes this even more fragile: any external `SIGTERM` to the watchdog group leaves the backend unsupervised.
    The fix is to make the cap a hard ceiling that cannot be silently disarmed:
    1. **Three-stage escalation**: `SIGINT` → wait `TG_SESSION_GRACE` (default 15 s) → `SIGTERM` → wait another `TG_SHUTDOWN_GRACE`-style window (5–10 s is enough) → `SIGKILL`. The final `kill -KILL` guarantees process termination regardless of backend cooperation. Apply the same `pkill -P $_dvb_pid` (process group kill) treatment used elsewhere so child processes (`tee`, model HTTP retries, npm wrappers) die with the parent.
    2. **Self-heal the watchdog**: replace `wait $! 2>/dev/null || exit 0` with a loop that retries the `wait` until either the cap elapses or the backend is observably gone. The watchdog must not exit early just because it received a signal — it should only exit when (a) the backend pid is dead, or (b) it has finished its escalation sequence and the parent script is responsible for moving on.
    3. **Hard wall-clock fallback**: in addition to the per-iteration `kill -0` checks, capture `_sweep_start_epoch=$(date +%s)` once and exit the watchdog loop if `(now - start) > sweep_max_session + 60` regardless of internal state. This is belt-and-suspenders against any pathological clock-skew or signal-storm scenario where the per-iteration math goes wrong.
    4. **Log marker on escalation**: emit `[HH:MM] sweep_watchdog escalation=SIGKILL pid=$_dvb_pid elapsed=Ns` so post-mortems can prove the watchdog actually fired. Today there is no log line that distinguishes "watchdog killed the backend at the cap" from "backend died on its own at the cap" — the only artifact is the eventual `sweep_done exit=143`. The `taskgrind-2026-05-01-2245-dotfiles-37908.log` reproduction shows zero watchdog log lines between 03:56 and 11:24, which is itself the smoking gun.
    5. **Apply the same fix to the regular session watchdog**. The regular per-session watchdog around `_dvb_timeout_pid` has the same shape and the same gap. Both `MAX_SESSION` and `SWEEP_MAX` cap-violation reproductions are plausible from the same root cause.
    Counter-arguments: (a) "SIGKILL is destructive — the backend may be mid-commit and a forced kill loses work" — the cap already implies the backend is not progressing; the existing `SIGINT`/`SIGTERM` window gives 15+ seconds for graceful shutdown before `SIGKILL`, which is more than enough for any well-behaved tool to checkpoint. A backend that needs 30+ minutes to respond to a signal has bigger problems than this watchdog can fix; (b) "the operator can already `SIGTERM` the grind manually" — yes, but the grind ran unsupervised overnight from 03:56 to 11:24; nobody was at the keyboard for 7 hours, and the cap exists precisely to prevent that scenario; (c) "process-group kill might hit `tee` or `caffeinate` and break logging" — addressed by killing only `$_dvb_pid` and its direct children via `pkill -P`, not the full session leader.
    **What's NOT in scope**: changing the cap defaults; reworking `pause_timer_for_network` (separate concern, time-based pause is fine when the watchdog itself is healthy); refactoring the watchdog into a separate Bash script (the inline watchdog is small enough that a 30-line patch fits inside `bin/taskgrind`); adding telemetry beyond the new escalation log line; trying to detect "is the backend wedged" mid-session — the cap fires at the budget, regardless of why.
    **Why P1**: this is the worst observed budget burn in the 2026-05 grind logs — a single wedged sweep ate the equivalent of 4 productive sessions. The fix is small, local, well-tested in shell pattern (process-group escalation is standard), and prevents an entire class of "grind runs all night and ships nothing" failure modes. P0 would require evidence of data-loss; this is "compute is wasted on wedged backends" which is severe but not corrupting.
  - **Files**:
    - `bin/taskgrind` — sweep watchdog (`bin/taskgrind:4312-4334`) and per-session watchdog (search for `_dvb_timeout_pid` initializer at `bin/taskgrind:2758` and its usage); add the escalation sequence and the `_sweep_start_epoch` hard fallback; emit the `sweep_watchdog escalation=...` log line
    - `tests/session.bats`, `tests/logging.bats` — at least three new tests: (a) fake backend that ignores `SIGINT` and `SIGTERM` is killed by the watchdog within `cap + grace + 5 s`, (b) watchdog continues to enforce the cap even after a spurious `SIGTERM` to the watchdog itself, (c) `sweep_watchdog escalation=SIGKILL` log line is emitted exactly once on the kill path
    - `tests/test_helper.bash` — extend the fake-backend fixture to support a `_FAKE_DEVIN_IGNORE_TERM=1` knob so tests can simulate the wedged-backend case without spawning a real claude-code process
    - `man/taskgrind.1` — TROUBLESHOOTING entry: "If a sweep ran past `TG_SWEEP_MAX`, check the log for `sweep_watchdog escalation=SIGKILL`. Absence of that line means the watchdog was disabled before the cap; file a bug."
    - `docs/architecture.md` — short note in the watchdog section: "The watchdog escalates `SIGINT` → `SIGTERM` → `SIGKILL` and uses a wall-clock fallback so a wedged backend cannot run past the cap."

- [ ] Session-watchdog must escalate to `SIGKILL` so a single claude-code session cannot run 4× past `TG_MAX_SESSION`
  - **ID**: session-watchdog-escalate-to-sigkill
  - **Tags**: watchdog, deadline, signals, claude-code, log-mining, AIFN-720-not-applicable
  - **Source**: `taskgrind-2026-05-01-1850-taskgrind-30444.log`. Session 15 (claude-code backend after `backend_rotated from=devin to=claude-code reason=zero_ship_streak`) ran from 04:55 to 11:24 — `duration=23323s tasks_after=45 shipped=0` — against `TG_MAX_SESSION=5400` (90 min). That is **6 h 29 m, or 4.32× over the cap**. The session output ends with `SessionEnd hook [bash ${CLAUDE_PLUGIN_ROOT}/hooks/scripts/hook.sh] failed: Hook cancelled`, suggesting the backend was wedged inside its own shutdown hook when something (apparently the deadline-driven graceful shutdown at 11:24) finally interrupted it. The session-watchdog never escalated to `SIGKILL`; if it had, the loss would have been bounded at ~95 min instead of ~390 min.
  - **Details**: This is the session-cap analogue of the existing `sweep-watchdog-escalate-to-sigkill` task — the same SIGINT → SIGTERM → (no SIGKILL) gap, but on the per-session watchdog at the `_dvb_session_timeout_watchdog` (or equivalent) site instead of the sweep watchdog. Apply the same fix pattern:
    1. **Three-stage escalation**: `SIGINT` → wait `TG_SESSION_GRACE` (default 15 s) → `SIGTERM` → wait an explicit `TG_SESSION_KILL_GRACE` window (default 10 s; new var) → `SIGKILL`. The final `kill -KILL` (plus `pkill -P "$pid"` for the session process group) guarantees termination even if the backend is stuck inside an unresponsive hook (claude-code SessionEnd hook in this case) or HTTP retry loop.
    2. **Self-heal the watchdog**: replace any `wait ... 2>/dev/null || exit 0` that lets the watchdog exit silently on signal interruption with a loop that retries `wait` until either the backend pid is dead or the escalation sequence has completed. The watchdog must outlive a spurious `SIGTERM` to the watchdog group itself.
    3. **Mark every escalation step in the log** so operators can tell *why* a session ran past its cap: `session_watchdog escalation=SIGINT pid=N elapsed=Ns`, `session_watchdog escalation=SIGTERM pid=N elapsed=Ns`, `session_watchdog escalation=SIGKILL pid=N elapsed=Ns`. - [ ] Stop the preflight "Git remote unreachable" warning from false-alarming on every macOS bash 3.2 run
  - **ID**: preflight-remote-watchdog-bash32-bug
  - **Tags**: preflight, bash-32, ux, false-alarm, git
  - **Details**: Every taskgrind run on macOS — the platform the repo's bash 3.2 compatibility guard explicitly preserves — prints `⚠ Git remote unreachable (may affect between-session sync)` even when `git ls-remote origin HEAD` returns the SHA in ~300ms with rc=0 (verified on this repo's own `origin` against `github.com/fyodoriv/taskgrind.git`). Confirmed root cause sits in the inline timeout watchdog at `bin/taskgrind:1162-1175`, with a byte-identical copy at `bin/taskgrind:1260-1273` inside `preflight_check_target`. The pattern is `(git ls-remote ... & _ls_pid=$!; sleep 10 & _sleep_pid=$!; (wait "$_sleep_pid"; kill "$_ls_pid") & _timer=$!; wait "$_ls_pid"; ...)`. The watchdog subshell does `wait "$_sleep_pid"` on a process that is its *sibling*, not its child — and macOS `/bin/bash` 3.2.57 refuses to `wait` on non-children. It returns 127 immediately, then proceeds to `kill "$_ls_pid"`. The still-running `git ls-remote` gets SIGTERM'd milliseconds after fork, the outer `wait "$_ls_pid"` returns 143 (128+SIGTERM), the if-condition fails, and the warning fires unconditionally. Reproduced cleanly with a 12-line `/bin/bash -c` minimal repro that consistently prints `outer rc=143` even though the wrapped command would have exited 0; bash 4+ (which the Linux CI runner uses) accepts `wait <any-pid>` for siblings, which is why the same code masks the bug under `ubuntu-latest`. The bug is purely in the timeout choreography, not the network call.
    Fix candidates, ranked:
    1. **Drop the hand-rolled watchdog; use git's own timeout knobs.** `GIT_HTTP_LOW_SPEED_LIMIT=1 GIT_HTTP_LOW_SPEED_TIME=10 git ls-remote origin HEAD` self-aborts at the protocol layer. No subshell pids, no `wait` semantics to get wrong, works unchanged on bash 3.2. Cleanest. Doesn't bound `git://` or `ssh://` connect-stage hangs, but those are rare for the personal-account remotes this tool actually targets, and they're already loosely bounded by the network-connectivity preflight check earlier in the same function.
    2. **Detect `gtimeout` (coreutils via Homebrew) and `timeout` (GNU/Linux), fall back to no-timeout when neither is available.** Honest about what the platform provides. Slightly more code than (1), but matches what other shell tools in the org do for the same problem.
    3. **Replace the buggy `wait` watchdog with a polling loop.** `for _ in $(seq 1 100); do sleep 0.1; kill -0 "$_ls_pid" 2>/dev/null || break; done; kill "$_ls_pid" 2>/dev/null`. Portable across bash 3.2 and 4+, but uglier than (1) and adds a 100ms granularity floor.
    4. **Drop the timeout entirely.** Defensible — git has reasonable connect timeouts via `GIT_HTTP_CONNECT_TIMEOUT` and OpenSSH `ConnectTimeout` defaults — but it loses the "preflight is bounded" invariant the rest of the function maintains, so probably not the right call.
    Recommend (1). Smallest diff, the env-var pair makes it obvious to the next reader *why* the timeout is there, and it doesn't add a platform-detection branch. Both call sites should land as a single helper (e.g. `_preflight_check_remote_reachable`) so the bug can't drift back in by being copy-pasted a third time.
