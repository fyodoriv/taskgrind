## P2

- [ ] Share immutable per-file fixtures via `setup_file()` so the bats suite stops paying full setup cost on every test
  - **ID**: bats-setup-file-shared-fixtures
  - **Tags**: tests, perf, bats, infrastructure
  - **Details**: `tests/test_helper.bash` currently runs the full setup body **for every one of 798 tests** â€” `mktemp -d`, 4Ă— `mkdir -p`, 2Ă— `cp` of `lib/constants.sh` + `lib/fullpower.sh`, write fake-devin, `chmod +x`. The recent perf commit (`4eb6ff4`) cut some of that cost via APFS clonefile (`cp -c`), but the *fundamental* duplication â€” re-running the same immutable setup 798 times when bats has explicit per-file lifecycle hooks â€” is still on the table. Bats has supported `setup_file()` and `teardown_file()` since version 1.4 (released 2021), and `BATS_FILE_TMPDIR` is auto-created/cleaned per file as the coordination point. Today this repo defines neither hook anywhere (`grep -lE 'setup_file' tests/*.bats` returns empty). The biggest concrete wins on top of today's work:
    1. **Lib-fixture sharing across tests in a file** â€” build the `dotfiles/lib/` tree once in `setup_file()`, then per-test `cp -Rc` (or fall back to `cp -R`) it into `$TEST_DIR`. Saves ~5-10s wall time on full suite per the dotfiles audit research already in `4eb6ff4`'s commit message.
    2. **Git template repo for `tests/git-sync.bats`** â€” the file does ~14 full `git init + config user.email + config user.name + commit --allow-empty` sequences (each ~200-400ms on macOS APFS). Building one template repo in `setup_file()` and `cp -Rc` cloning it per-test should drop that to <50ms total. Estimated 10-15s saved on full suite at jobs=8.
    3. **Fake-devin stub sharing** â€” most tests in the suite use the exact same "record args and exit 0" stub. Build it once in `setup_file()`, symlink per-test (or just `export DVB_GRIND_CMD` to the shared path).
    Implementation pattern (verified against bats-core 1.10 docs): define `setup_file()` directly in `tests/test_helper.bash` so any file that does `load test_helper` picks it up automatically; bats runs setup_file in a different bash subshell from per-test `setup()`, so coordination must go through files in `BATS_FILE_TMPDIR` (env-var exports won't propagate). Backwards-compatible: tests that already work keep working, the new path is the fast lane.
    **Risks** the implementer should know about: (a) `setup_file()` runs *before* the first `setup()` â€” if any test mutates the shared fixture it'll break later tests in the same file (use `cp -R` per test, never link to it directly); (b) bats `--no-parallelize-within-files` flag MUST stay off (today's measurement showed it triples runtime on this suite); (c) the git template repo trick assumes the repo's git config is what tests actually want â€” any test that needs a different config (e.g. a different default branch) needs to override post-clone, not at template build.
    **Why P2, not P1**: today's `4eb6ff4` already shipped the highest-ROI free wins (jobs=8, ulimit, awk, clonefile). Further speedup costs implementation time on a non-broken system. Operators benefit (faster `make check` cycle) but no one is blocked.
  - **Files**: `tests/test_helper.bash` (new `setup_file()` + supporting helpers like `_taskgrind_clone_template`); `tests/git-sync.bats` (use the new template path); optionally `tests/session.bats`, `tests/signals.bats` if they show up as the next bottleneck after the first round; `AGENTS.md` (note the new pattern under "Local Test Notes" so future bats files use it from the start)
  - **Acceptance**: `time make test-force` on a controlled clean machine shows â‰Ą15% wall-time reduction vs the post-`4eb6ff4` baseline (don't accept "looks faster" without a number); the implementation works on Linux too (no APFS-only assumptions baked in); `make check` passes locally and the GHA test jobs go green at the same rate they did before (i.e. no new flakes introduced â€” pair with the `ci-pre-existing-test-failures` task if green CI is a prerequisite); a one-paragraph note in `AGENTS.md` "Local Test Notes" documents the `setup_file` + `BATS_FILE_TMPDIR` pattern so the next bats author doesn't reinvent it; the new helper has its own doc-drift test in `tests/basics.bats` (or a new `tests/test-helper.bats`) that asserts `setup_file` is defined and the cloning helper falls back gracefully when `cp -c` is rejected

## P3

- [ ] Tag tests by speed class and add a `make test-fast` target so the dev iteration loop is sub-30s
  - **ID**: make-test-fast-target
  - **Tags**: tests, dx, bats, tagging
  - **Details**: Today the only choice is `make test` (full suite, currently ~1-3 min depending on system load) or `make test TESTS=tests/<file>.bats` (single file, requires the dev to know which file). Neither is the right interactive cycle for the common case of "I'm editing `bin/taskgrind` and want to know in <30s whether I broke a structural invariant or just a behavioural test." Bats-core 1.x supports `# bats test_tags=tag1,tag2` decorators and `--filter-tags` selection ([bats-core/bats-core#519](https://github.com/bats-core/bats-core/pull/519), shipped in 1.5+). Concretely:
    - Audit `tests/*.bats` and tag every test as either `fast` (structural / grep-based / no `mktemp`-and-launch-bin) or `slow` (everything else: anything that runs `"$DVB_GRIND"`, anything in `signals.bats` / `session.bats` / `multi-instance.bats` / `resume.bats`, anything calling `_wait_for_file_pattern`, and the `make audit`-invoking tests in `basics.bats` since shellcheck is genuinely slow). Rough estimate from `wc -l` + spot-checks: ~300 fast, ~498 slow. The fast subset on a controlled run should clear in 20-30s on Apple Silicon at the new jobs=8 cap.
    - Add Makefile targets: `make test-fast` runs `bats --jobs $(TEST_JOBS) --filter-tags fast tests/*.bats`; `make test-slow` runs the inverse with a lower jobs cap (4) since the slow tests are the ones that race on parallelism. `make test` and `make check` keep their current behaviour (run everything).
    - Update README.md "Development" + AGENTS.md "Development" sections so the dev workflow points to `make test-fast` first.
    - Optionally: a CI matrix split â€” `test-fast` job runs on every push and gates merges; `test-slow` job runs nightly or on a `[ci-full]` commit-message marker. Out of scope here unless the CI parallel-load flakes from `ci-pre-existing-test-failures` are killed first; otherwise this just adds a second flaky job.
    **What this is NOT**: a justification for letting test count drift, or a reason to skip writing slow tests. Both subsets must keep passing on every push to main; the only difference is which one runs in the dev's tight loop. The bats `--filter-tags` engine is well-supported and stable; this is not exotic infrastructure.
    **Why P3**: pure developer-experience optimization. Today's `1-3 min` test cycle isn't broken â€” it's just not as tight as it could be. Operators who land here after the P1 (CI green) and P2 (setup_file fixtures) work get a smaller incremental win.
  - **Files**: every `tests/*.bats` (add the `# bats test_tags=...` line above each `@test`); `Makefile` (new `test-fast` and `test-slow` targets); `README.md` (Development section); `AGENTS.md` (Development section + Local Test Notes); optionally `.github/workflows/check.yml` if the CI matrix split is in scope
  - **Acceptance**: `make test-fast` finishes in â‰¤30 seconds on a clean Apple Silicon machine (measured, not estimated); `make test-fast` passes 100% (the fast subset is by definition the deterministic one â€” no flakes); `make test-slow` runs the rest and matches the current `make test` exit status when the suite is otherwise green; the README/AGENTS guidance points devs at `make test-fast` for tight loops; a doc-drift test in `tests/basics.bats` asserts the new Makefile targets are documented in both README and AGENTS so they don't silently disappear; the tag audit passes a sanity check (no test is both `fast` and `slow`; every test has at least one tag)
